# Tiny Eval

A tiny harness for evaluating short factual question answering against
Claude. Reads a directory of `*.eval.md` files (YAML frontmatter +
markdown body), calls the model once per eval, grades exact-match first
and falls back to a model-as-judge call only on miss, and writes a
single Markdown report.

## One-command run

From the repo root:

```bash
npm install
ANTHROPIC_API_KEY=sk-ant-... npm run eval
```

That's it. The defaults read from `artifacts/tiny-eval/evals/` and
write `artifacts/tiny-eval/report.md`. With the 5 sample evals in this
repo, a full run uses ~1.5k input tokens and ~30 output tokens total
(answer + judge combined). On Sonnet 4.5 + Haiku 4.5 list pricing that
is well under **$0.01 per run** — the kind of cost you can ignore.

## Files

- `evals/*.eval.md` — one eval per file. Frontmatter must include
  `prompt:` and `expected:`; everything below the closing `---` is the
  "why this eval exists" body and ends up in the rendered report.
- `report.md` — generated by `npm run eval`. Not committed.
- `../../scripts/eval-run.ts` — the harness itself.

## Eval file shape

```markdown
---
category: easy
prompt: 'What is the chemical symbol for gold?'
expected: 'Au'
---

<!-- why this eval: one-liner — first sentence is the durable intent. -->

Longer body here is optional; it ends up in the report directly under
the eval's heading, so future readers see _why_ this eval exists, not
just whether it passed.
```

Frontmatter rules (see `parseFrontmatter` in `scripts/eval-run.ts`):

- file MUST start and end the frontmatter block with `---` on its own line
- one `key: value` per line, no nesting, no list values
- `"..."` and `'...'` quoting is honored; inside double quotes you can use
  `\n`, `\t`, `\"`, `\\`
- only `prompt` and `expected` are required; `category` is optional and
  shown in the report

## CLI flags

```
npm run eval -- [--dir DIR] [--out FILE] [--model NAME] [--judge NAME] [--max-tokens N]
```

| flag           | default                         | meaning                                        |
| -------------- | ------------------------------- | ---------------------------------------------- |
| `--dir`        | `artifacts/tiny-eval/evals`     | where to find `*.eval.md`                      |
| `--out`        | `artifacts/tiny-eval/report.md` | report path                                    |
| `--model`      | `claude-sonnet-4-5`             | answer model id                                |
| `--judge`      | `claude-haiku-4-5-20251001`     | judge model (only invoked on exact-match miss) |
| `--max-tokens` | `256`                           | upper bound on answer + judge output tokens    |

## How grading works

1. **Exact match.** Both strings are case-folded, whitespace-collapsed,
   and stripped of trailing `.!?`. If they match, the eval passes and the
   judge is **not** called.
2. **Judge fallback.** If exact match fails, the judge model gets the
   prompt, the expected answer, and the model answer, and replies with
   `VERDICT: pass|fail` + a one-sentence reason. The reason is rendered
   in the report so disagreements are inspectable.

This is intentionally a stupid grader. It works for short factual QA;
for anything richer you want a real eval framework.

## Env

- `ANTHROPIC_API_KEY` — required. The harness fails fast if it's missing.

## Cost note

A full run with the 5 sample evals is ~1.5k input + ~30 output tokens.
At Sonnet 4.5 + Haiku 4.5 list pricing that is under one US cent per
run. Running it on every commit would be free; running it on every CI
build would still be free. We keep `--max-tokens` capped at 256 so a
runaway answer can't blow that budget by an order of magnitude.

## What this is not

- not parallel — evals run sequentially
- not retried — one shot per eval, transient errors fail the run
- not multi-model — one answer model + one judge model, period
- not embedding-graded — exact + LLM-judge only
- not a web UI — Markdown report, that's it

These are explicit non-goals from the spec. If you need any of them,
this is not the artifact you want.
